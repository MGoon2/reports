요약

  1. 사실: 이 프로젝트는 ASP.NET Core/Generic Host가 아니라 WPF WinExe다. Program.cs, Startup.cs, CreateHostBuilder, WebApplication.CreateBuilder 흐름은 확인되지 않았고, App.xaml의 Startup="Application_Startup"이 진입점이다. 근거: FinUp.CollectData.App.Crawler.News.csproj:4-6, App.xaml:5.
  2. 사실: appsettings.json은 시작 시 필수 로드되고 Serilog 설정에만 연결된다. 크롤러 동작 옵션은 대부분 App.config를 ConfigurationManager로 직접 읽는다. 근거: App.xaml.cs:30-38, App.xaml.cs:58-68, AppSetting.cs:10-22.
  3. 사실: DI 등록은 메모리 캐시, CacheRepository, MainViewModel, MainWindow, Debugger singleton에 한정된다. 핵심 크롤링/Kafka/DB 객체는 ViewModel과 Biz 클래스 내부에서 직접 생성된다. 근거: App.xaml.cs:52-73, MainViewModel.cs:50-87.
  4. 사실: 실행 후 MainViewModel은 1초 DispatcherTimer, 형태소 Kafka consumer, 디버그 Kafka consumer, 3초 ping producer를 시작한다. 근거: MainViewModel.cs:52-93.
  5. 사실: 주기 도달 시 CrawlerSiteRepository.Run이 DB에서 활성 사이트 목록을 가져오고, 사이트별 병렬 요청/파싱/필터링/Mongo 저장/Kafka 또는 Telegram 발송을 수행한다. 근거: CrawlerSiteRepository.cs:84-121, 171-400, 418-628.
  6. 추론: 설정 변경 영향은 값마다 다르다. AppSetting static initializer로 읽은 값은 최초 접근 후 고정되고, getter 기반 시간/필터 값은 호출 시 다시 읽힐 가능성이 있으나 소비 클래스가 생성 시 캐싱하면 즉시 반영되지 않는다.
  7. 사실: 종료는 MainWindow.OnClosed에서 Application.Current.Shutdown()을 호출하는 수준이다. consumer/task/timer/provider dispose 또는 cancellation token 연결은 확인되지 않는다. 근거: MainWindow.xaml.cs:33-37, Consumer.cs:79-104, Consumer.cs:120-153.

옵션별 상세 분석

설정값 변경 시 런타임 영향

  • 사실: appsettings.json은 reloadOnChange=true이나, Serilog 외 옵션은 주석 처리된 Configure 코드에만 연결되어 있다. 근거: App.xaml.cs:32, App.xaml.cs:58-68.
  • 추론: 실행 중 appsettings.json의 CrawlerConfig, TelegramConfig, ConnectionStrings를 바꿔도 현재 크롤러 동작은 바뀌지 않는다.
  • 사실: AppSetting의 DB/Kafka/Proxy/Telegram 값 일부는 static initializer로 읽는다(AppSetting.cs:10-22, 49-61). 이런 값은 최초 접근 후 고정된다.
  • 사실: 시간/필터 값 다수는 getter에서 OpenExeConfiguration 또는 AppSettings를 다시 읽는다(AppSetting.cs:65-677). 단 HtsFilter처럼 생성 시 field에 복사하면 이후 변경 반영이 제한된다(HtsFilter.cs:25-26).

전송 토글

SendNewsCrawlerTelegram은 CrawlerUtil.SendMessageList의 Telegram 전송을 막는다(CrawlerUtil.cs:131-132). SendNewsCrawlerKafka는 같은 유틸 경로의 Kafka 발행에만 적용된다(CrawlerUtil.cs:123-129). 하지만 HTS/KoreaBriefing을 형태소 분석기로 보내는 CrawlerSiteRepository.SendKafka 경로는 이 토글과 독립적으로 실행된다(CrawlerSiteRepository.cs:531-536, 574-578, 697-711).

시간 창

크롤링 시간은 site별 요청 전에 검사된다(CrawlerSiteRepository.cs:178-183, CrawlerUtil.cs:244-280). 발송 시간은 메시징 단계에서 검사된다(CrawlerSiteRepository.cs:523-628, CrawlerUtil.cs:202-242). Debug 빌드에서는 GetWorkTime이 즉시 true를 반환한다(CrawlerUtil.cs:186-190, CrawlerSiteRepository.cs:936-940).

DB 기반 동적 구성

활성 크롤링 대상은 파일이 아니라 DB의 T_CrawlerSite에서 온다(SqlCrawlerDac.cs:8-27, CrawlerSiteRepository.cs:95-108). 필터 단어와 언론사/이미지 필터도 DB에서 조회한다(BaseFilter.cs:18-24, HtsFilter.cs:32-46, SqlCrawlerDac.cs:61-73, 151-200).

전체 실행 플로우

  1. WPF WinExe 프로세스 시작 -> App.xaml Startup 이벤트 호출.
  2. Application_Startup에서 appsettings.json 로드 및 Serilog 초기화.
  3. ServiceCollection 등록 후 BuildServiceProvider 실행.
  4. MainWindow resolve/show. MainViewModel 생성 및 DataContext 설정.
  5. MainViewModel.Init: TimeDuration 초기화, CrawlerSiteRepository 생성, DispatcherTimer 시작.
  6. CrawlerSiteMorpheme.Init: Kafka consumer, Mongo/SQL DAC, 형태소 중복 캐시 준비 후 HTS_NEWS_PROCESS consume task 시작.
  7. DEBUGGER_PONG consumer와 3초 주기 DEBUGGER_PING producer task 시작.
  8. DispatcherTimer tick마다 TimeDuration 감소. 0이고 RUNNING이 아니면 operation history START 기록 후 Task.Run으로 _biz.Run 실행.
  9. CrawlerSiteRepository.Run: StockData.SetStockData, QCrawlerSiteList, Mongo collection 준비, siteList 생성.
  10. CrawlingParallel: 채널별 bag 생성, 휴장일 조회, MaxDegreeOfParallelism=10으로 사이트별 처리.
  11. 사이트 처리: crawl 시간 검사, TypeContents에 따라 XML/HTML/JSON 요청, Type switch로 전용 crawler GetNews 호출.
  12. 필터링: HTS/KoreaBriefing/SpecialStock 필터 적용, 원본 목록 Mongo InsertMany.
  13. 메시징: 휴장일/발송 시간 검사 후 Kafka 형태소 topic 또는 CrawlerUtil.SendMessageList 호출.
  14. 형태소 consumer: base64 decode, JSON parse, Mongo morpheme update, 중복 형태소 필터, CrawlerUtil.SendMessageList로 발송.
  15. 성공 시 operation history STOP, 예외 시 ERROR 기록. 창 닫힘 시 Application.Current.Shutdown 호출.

  • 사실: 메인 창이 닫히면 MainWindow.OnClosed가 Application.Current.Shutdown()을 호출한다. 근거: MainWindow.xaml.cs:33-37.

리스크와 확인 필요 사항

appsettings.json의 ConnectionStrings/TelegramConfig/CrawlerConfig는 실제 런타임 크롤러 설정이 아니다. 운영자가 이 파일만 수정하면 효과가 없을 가능성이 높다. 근거: App.xaml.cs:58-68.
AppSetting static initializer가 많은 필수 키를 직접 읽는다. 누락/형식 오류가 실제 사용 전 앱 시작 경로를 중단할 수 있다. 근거: AppSetting.cs:10-22,49-61.
CoinAlarmStartTime/CoinAlarmEndTime/GameFocusAlarmStartTime/GameFocusAlarmEndTime은 코드가 읽지만 App.config에서는 주석 처리되어 있다. Release에서 해당 발송 경로 실행 시 예외 가능. 근거: App.config:95-98; AppSetting.cs:435-494; CrawlerSiteRepository.cs:602-615.
Parallel.ForEach에 async lambda가 사용된다. EDaily.GetNewsAsync 같은 await 작업을 Parallel.ForEach가 제대로 기다리지 않는 구조일 수 있다. 근거: CrawlerSiteRepository.cs:171,373-376.
Kafka/Telegram 발송이 여러 곳에서 fire-and-forget Task.Run으로 시작되어 Run 완료/operation history STOP보다 늦게 끝날 수 있다. 근거: CrawlerSiteRepository.cs:531-536,574-590,598-623.
형태소 캐시 prewarm은 Morpheme == null 조회 후 Morpheme == null이면 continue 하는 구조라 기존 형태소를 캐시에 넣지 못할 가능성이 있다. 근거: CrawlerSiteMorpheme.cs:73-103.
window close 외 graceful shutdown 경로가 없다. consumer loop/timer/provider dispose가 확인되지 않는다. 근거: MainWindow.xaml.cs:33-37; Consumer.cs:79-153.
App.config/appsettings.json에 DB/Kafka endpoint, Telegram token, proxy credential 등 민감 설정이 들어 있다. 이 보고서에서는 값을 마스킹했다.

검증 및 한계

  • 코드 파일은 수정하지 않았다. 생성한 파일은 이 HTML 보고서뿐이다.
  • 앱 실행/빌드는 수행하지 않았다. 빌드는 bin/obj 갱신 가능성이 있어 읽기 전용 분석 목적과 분리했다.
  • 외부 DB의 실제 사이트/필터 데이터, Kafka topic 상태, Telegram API 응답은 저장소만으로 확인할 수 없다.
  • 중요 주장은 파일 경로와 라인 번호 근거를 함께 표기했다.
