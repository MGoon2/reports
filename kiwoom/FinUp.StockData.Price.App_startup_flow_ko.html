핵심 결론

실제 StartUp 옵션의 기준은 주석이 아니라 Program.cs의 if/else 체인이다. Program.cs:20-48에서 총 14개 값이 폼 또는 스케줄러로 매핑된다.

  • 현재 설정은 App.config:75의 StartUp=KR이며, 실행 시 SchedulerKiwoomReal이 자동 로드된다.
  • KR, KT, X, M은 폼 로드 시 자동 스케줄을 시작하는 성격이다.
  • FC, F, FO, CC, XO, EMI, KMI, KBM, T는 폼을 띄운 뒤 버튼으로 실행하는 수동/운영 도구 성격이 강하다.
  • S는 원래 Xing TR 종합 스케줄러로 보이나 현재 SchedulerXingTR.cs:81-83에서 InitSchedule() 호출이 주석 처리되고 즉시 return하므로 자동 스케줄은 비활성 상태다.

런타임 리스크: App.config에 KiwoomStartDelay가 91행과 100행에 중복 정의되어 있다. .NET Framework의 appSettings 중복 키는 설정 로딩 오류 또는 적용값 불명확성을 만들 수 있으므로 KR 실행 전에 정리해야 한다.

Kafka, Redis/Valkey, MongoDB 전환 관점

  • Kafka: StartUp 분기에서 Kafka 작업을 직접 시작하는 코드는 확인되지 않았다. Kafka는 로그 하위 시스템 설정인 UseLog, LogCommunication=kafka, KafkaUrl 쪽 의존성이다. 현재 UseLog=false라 실행 경로상 비활성으로 보는 것이 타당하다.
  • Redis/Valkey: 현재 Redis 엔드포인트는 Valkey 성격의 AWS ElastiCache 주소로 설정되어 있다. 하지만 현재 KR/PRICE 설정에서는 RedisDistribute=false라 가격 배포가 동작하지 않는다. S의 TickRedis 같은 일부 수동/스케줄 작업은 Redis 캐시를 사용할 수 있다.
  • MongoDB: KR에서는 현재 MongoSaveData=false라 가격 저장이 꺼져 있다. 반면 X, M, KBM, T는 MongoDB가 핵심 저장소/이관 대상이다. PG 전환 범위는 KR보다 X/M/KBM/T 쪽이 더 직접적이다.

정리 및 권고

  1. StartUp 허용값을 Program.cs 기준으로 문서화하고, App.config:72의 오래된 주석은 제거 또는 갱신한다.
  2. KiwoomStartDelay 중복 키를 먼저 정리한다. 현재 KR 플로우의 시작 시각 계산에 직접 영향을 준다.
  3. 운영 자동 실행 후보는 KR, KT, X, M 네 가지로 우선 분류한다. 나머지는 운영자가 버튼을 눌러 실행하는 도구로 분리하는 것이 안전하다.
  4. MongoDB를 PG로 전환할 때는 KR의 옵션성 Mongo 저장보다 X의 실시간 임시 저장, M의 이관, KBM의 복원/분봉 재생성, T의 수동 이관 도구를 먼저 영향 분석해야 한다.
  5. Kafka 제거는 실행 경로보다 프로젝트 참조/로그 라이브러리 정리 이슈다. UseLog=false 운영이 전제라면 런타임 영향은 낮지만, 빌드 참조와 설정 키 제거는 별도 작업으로 검증해야 한다.
