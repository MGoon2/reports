Executive Summary

The authoritative StartUp option list is the if/else chain in Program.cs, not the stale comment in App.config. Program.cs:20-48 maps 14 option values to WinForms entry points.

  • The current setting is StartUp=KR at App.config:75, which starts SchedulerKiwoomReal.
  • KR, KT, X, and M start automatic schedules on form load.
  • FC, F, FO, CC, XO, EMI, KMI, KBM, and T are mostly manual operator tools.
  • S looks like the Xing TR scheduler, but its automatic schedule is currently disabled by an early return after market-info loading.

Runtime risk: KiwoomStartDelay is declared twice in App.config at lines 91 and 100. This can cause configuration loading errors or ambiguous effective behavior in .NET Framework app settings.

Automatic Scheduler Details

KT - Kiwoom TR Trader Day

SchedulerKiwoomTR initializes on load, schedules TraderDayKiwoom("거래원일별") for market close plus 3.5 hours, and restarts the process on completion. Evidence: SchedulerKiwoomTR.cs:43-49, 96-143, 158-165.

X - Xing Real-time Hoga/Trader

SchedulerXingReal logs into Xing and collects stocks before market open, then starts Job.XingReal.HogaTrader. The job creates daily MongoDB temp collections such as T_Hoga_yyyyMMdd and T_Trader_yyyyMMdd. Evidence: SchedulerXingReal.cs:43-62, 94-199, 214-232, Job.XingReal\HogaTrader.cs:57-87, 279-325.

M - Hoga/Trader Migration

SchedulerHogaTraderMove schedules HogaTraderMove at 23:55 for the next trading day context. The job copies MongoDB data from StockDataTemp daily collections into StockData.T_Hoga and StockData.T_Trader. Evidence: SchedulerHogaTraderMove.cs:24-89, Job.XingReal\HogaTraderMove.cs:30-47, 58-163.

S - Xing TR Scheduler State

SchedulerXingTR currently returns before its schedule is initialized. If that early return is removed, InitSchedule() would schedule a broad TR workload: stock list, daily cache, validation, gap close, stock status, VI real-time, index real-time, daily candles, index, deposit trend, investor, and after-close jobs. Evidence: SchedulerXingTR.cs:67-83, 133-320.

Kafka, Redis/Valkey, MongoDB Notes

  • Kafka: No StartUp branch directly starts Kafka. Kafka is part of the logging subsystem controlled by UseLog, LogCommunication=kafka, and KafkaUrl. With UseLog=false, it is effectively inactive at runtime.
  • Redis/Valkey: The configured Redis endpoint is an AWS Valkey-style endpoint. In the current KR/PRICE flow, RedisDistribute=false disables price distribution. Some S jobs, such as TickRedis, can still use Redis when run.
  • MongoDB: KR currently disables Mongo price saving, but X, M, KBM, and T use MongoDB directly and should be prioritized for a PostgreSQL migration analysis.

Recommendations

  1. Document allowed StartUp values from Program.cs and update the stale App.config comment.
  2. Remove the duplicate KiwoomStartDelay key before relying on the KR schedule.
  3. Classify KR, KT, X, and M as production-style automatic schedulers; keep the rest as manual operator tools unless intentionally promoted.
  4. For MongoDB-to-PostgreSQL work, prioritize X, M, KBM, and T before the currently disabled Mongo branch in KR.
  5. Kafka removal is primarily a project reference and logging-library cleanup task rather than a StartUp runtime-flow change under the current UseLog=false setting.
