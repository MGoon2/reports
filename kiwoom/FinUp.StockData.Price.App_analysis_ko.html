요약 결론

Kafka

KR 런타임 기준 실사용 근거 없음. 설정에는 KafkaUrl, LogCommunication=kafka가 남아 있지만 UseLog=false이고 대상 앱 코드에서 FinUp.Core.Log.Logger 호출은 발견되지 않았다.

Redis / Valkey

코드는 Redis Pub/Sub 경로를 보유하지만 현재 KR/PRICE 설정에서는 자동 분배가 꺼져 있다. Valkey endpoint는 이미 설정되어 있으며, 활성화 시 stockdatapubsub 클러스터로 publish한다.

MongoDB / PG 전환

현재 KR/PRICE 설정에서는 Mongo 저장이 꺼져 있다. 더 강하게는 코드상 PRICE Mongo handler의 StartAsync 호출도 주석 처리되어 있어, 플래그만 켜도 저장이 시작되지 않는다.

따라서 현재 StartUp=KR, FIDType=PRICE, RedisDistribute=false, MongoSaveData=false 기준의 자동 실행 경로는 키움 실시간 가격 이벤트를 수신하지만 Kafka/Redis/MongoDB로 자동 송신 또는 저장하지 않는 구조로 판단된다.

Kafka 확인

Evidence

  • App.config에는 KafkaUrl, LogCommunication=kafka가 있다. 동시에 UseLog=false다. App.config:101-107
  • 대상 앱 코드에서 FinUp.Core.Log, Logger.Instance, LoggerKafkaDAL 직접 사용은 검색되지 않았다.
  • FinUp.Core.Data는 SQL 실행 후 Log.Insert를 호출하고, Log.Insert는 Logger.Instance.InsertManualy로 이어진다. Sql.cs:84-95, 173-184, Log.cs:16-62
  • Logger는 생성 시 UseLog가 true일 때만 LoggerFactory와 DAL을 만든다. Logger.cs:22-30
  • LoggerFactory는 LogCommunication=Kafka일 때 LoggerKafkaDAL을 생성하고, 이 DAL은 KafkaUtil로 Confluent.Kafka producer를 만든다. LoggerFactory.cs:50-57, LoggerKafkaDAL.cs:7-24, KafkaUtil.cs:31-64
  • 프로젝트 참조상 FinUp.Core.Log와 Confluent.Kafka 의존성은 빌드 그래프에 남아 있다. FinUp.StockData.Price.App.csproj:476-479, FinUp.Core.Log.csproj:89-91

Inference

KR 런타임 기준 Kafka는 기능적으로 비활성이다. 다만 빌드/배포 의존성으로는 FinUp.Core.Log 및 Confluent.Kafka가 남아 있으므로, “운영에서 Kafka를 호출하지 않는다”와 “Kafka 라이브러리/설정을 제거해도 된다”는 서로 다른 판단이다.

판단신뢰도설명
현재 KR 실행에서 Kafka 송신 없음높음UseLog=false가 Kafka DAL 생성을 차단하고 대상 앱 내 직접 호출이 없다.
Kafka 설정 삭제 가능중간런타임 호출은 없지만 Core.Data/Core.Log 빌드 의존성이 있어 제거 범위 확인이 필요하다.

Redis / AWS Valkey 확인

Evidence

  • stockdata, stockdatapubsub Redis cluster가 finup custom config section에 정의되어 있다. 현재 endpoint는 master.stockdatapubsub-valkey-dev...다. App.config:4-33
  • SchedulerKiwoomReal은 로드시 stockdatapubsub endpoint를 로그에 출력한다. 이 자체는 설정 접근이며 publish는 아니다. SchedulerKiwoomReal.cs:84-89
  • PRICE 경로에서는 RedisDistribute=true일 때만 RedisStockDataHandler가 생성된다. 현재 값은 false다. App.config:98, KiwoomRedis.cs:71-78
  • RedisStockDataHandler는 RedisDatabase.GetSubcriber("stockdatapubsub")로 subscriber를 얻고 PRICE, HOGA, TRADER channel에 publish한다. RedisStockDataHandler.cs:24-27, 31-51, 73-121
  • SchedulerKiwoomReal.button1_Click는 수동 테스트 버튼으로 Redis Pub/Sub 테스트 메시지를 publish한다. 자동 실행 경로와는 구분된다. SchedulerKiwoomReal.cs:344-380
  • Core.Redis는 StackExchange.Redis 2.1.58를 사용하고 ConfigurationOptions에 endpoint, timeout, retry 등을 넣어 연결한다. SSL/AUTH 관련 설정은 현재 schema에 없다. FinUp.Core.Redis.csproj:75-80, RedisCluster.cs:138-149, Configuration/FinUp.cs:140-214

Inference

현재 KR/PRICE 자동 실행에서 Redis publish는 꺼져 있다. Redis를 AWS Valkey로 전환하는 관점에서는 이미 endpoint가 Valkey dev로 잡혀 있으나, 실제 Redis Pub/Sub 기능은 RedisDistribute=true 또는 HOGATRADER 경로에서 검증해야 한다.

Valkey가 Redis 프로토콜 호환 모드이고 인증/TLS를 강제하지 않는 구성이라면 현재 StackExchange.Redis 경로로 동작할 가능성이 높다. 반대로 AWS Valkey에서 TLS 또는 AUTH token을 요구하면 현재 config schema에는 Ssl, Password, User를 표현할 곳이 없어 코드/설정 확장이 필요하다.

특이점: FIDType이 PRICE가 아니면 KiwoomRedis의 else 경로가 RedisDistribute 플래그를 보지 않고 RedisStockDataHandler를 생성한다. 현재는 FIDType=PRICE라 해당되지 않지만, HOGATRADER 운용 시 Redis는 사실상 강제 사용된다. KiwoomRedis.cs:87-98, 133-140

MongoDB / PostgreSQL 전환 확인

Evidence

  • App.config에는 MongoDB, MongoDBName=StockData가 남아 있다. App.config:76-81
  • PRICE 경로에서 MongoSaveData=true일 때만 MongoStockDataHandler를 생성한다. 현재 값은 false다. App.config:99, KiwoomRedis.cs:80-85
  • 그러나 KiwoomRedis.Start의 PRICE Mongo handler 시작 호출은 주석 처리되어 있다. 즉 플래그를 true로 바꾸는 것만으로는 PRICE Mongo 저장 consumer가 시작되지 않는다. KiwoomRedis.cs:126-131
  • HOGA Mongo handler 생성 및 시작도 주석 처리되어 있다. KiwoomRedis.cs:89-98, 133-140
  • MongoStockDataHandler는 DB명을 {MongoDBName}{yyyy}로 바꾸며, T_Hoga, T_Trader, T_Price collection을 만들고 InsertManyAsync로 batch insert한다. MongoStockDataHandler.cs:27-43, 60-83, 86-117
  • 프로젝트에는 마이그레이션/테스트/다른 StartUp 경로의 Mongo 직접 사용이 다수 있다. 예: Mig\MongoDBMig.cs, Mig\KiwoomBackupMig.cs, Job.v2\TraderDayKiwoom.cs, Test\TraderDay.cs. 이들은 KR 자동 가격 수집 경로와 별도다.

Inference

현재 KR/PRICE 운용만 놓고 보면 MongoDB는 즉시 제거 후보에 가깝다. 하지만 프로젝트 전체에서 마이그레이션 폼, 테스트 폼, 다른 스케줄러/StartUp 경로가 MongoDB에 직접 연결하므로 “앱 전체 MongoDB 제거”는 KR 경로보다 넓은 작업이다.

PG 전환 시 KR/PRICE 자동 수집만 목표라면 우선순위는 낮다. 반대로 과거/일봉/거래원/마이그레이션 기능까지 유지하려면 Mongo collection T_Price, T_Hoga, T_Trader, T_TraderDay 등과 연도 suffix DB 전략을 PG schema/table partition 전략으로 매핑해야 한다.

Unknowns / Limits

  • 실제 AWS Valkey endpoint 접속 가능 여부, TLS/AUTH 적용 여부는 저장소만으로 확정할 수 없다.
  • 배포 산출물에 Kafka 관련 DLL을 포함하는지, 운영 모니터링에서 Kafka traffic이 실제로 0인지까지는 빌드/런타임 관측이 필요하다.
  • PG 전환 범위가 KR 실시간 가격 수집만인지, 마이그레이션/테스트/다른 StartUp 경로까지 포함하는지는 별도 범위 결정이 필요하다.
  • App.config에는 KiwoomStartDelay가 중복 정의되어 있다. 현재 분석 주제는 아니지만 .NET configuration 로딩 정책에 따라 런타임 설정 오류 또는 예기치 않은 값 해석 가능성이 있어 별도 확인 가치가 있다. App.config:91, 100
