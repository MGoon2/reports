Executive Summary

With the uploaded executable config, StartUp=KR dispatches to SchedulerKiwoomReal, then KiwoomRedis, then Kiwoom real-time PRICE collection.

  • Redis is active. The uploaded config does not define RedisDistribute, and the code defaults missing values to true.
  • MongoDB is instantiated but not effectively saving PRICE data. Missing MongoSaveData defaults to true, but _mongoPriceHandler.StartAsync(...) is commented out.
  • Kafka is inactive at runtime for this config. UseLog, System, SystemType, and KafkaUrl are absent; LoggerConfigModel.UseLog returns false when System falls back to its anonymous default.
  • Operation logging is active. Missing UseOperationLog leaves the operation-log default as true for JobBase2 and AliveLogger.

Main difference from the repository App.config: the repository config explicitly sets RedisDistribute=false, MongoSaveData=false, and UseOperationLog=false. The uploaded executable config omits those keys, so the code defaults change behavior.

Conclusion

With the uploaded KiwoomReal executable config, the application performs Kiwoom real-time PRICE collection and Redis publishing is active. Kafka is inactive at runtime. MongoDB is the nuanced case: the handler is instantiated and receives price events, but the Mongo consumer is commented out, so data is not inserted and queued events may accumulate.

For migration work, Kafka removal has low runtime impact under this config, Redis/Valkey replacement is directly on the active path, and Mongo-to-PostgreSQL scope depends on whether the currently dormant PRICE save path should remain disabled or be reintroduced using PostgreSQL.
