핵심 결론

업로드된 실행 config 기준 실제 실행 분기는 StartUp=KR이며, SchedulerKiwoomReal → KiwoomRedis → Kiwoom 실시간 PRICE 수집 경로로 진행된다.

  • Redis는 사용된다. config에 RedisDistribute 키가 없고, 코드 기본값이 true이므로 PRICE Redis publish 핸들러가 생성 및 시작된다.
  • MongoDB는 연결 객체와 큐는 만들어지지만 실제 저장 소비자는 시작되지 않는다. config에 MongoSaveData 키가 없어 기본값은 true지만, KiwoomRedis의 _mongoPriceHandler.StartAsync(...) 호출이 주석 처리되어 있다.
  • Kafka는 이 실행 config 기준 런타임에서 비활성으로 판단된다. UseLog, System, SystemType, KafkaUrl이 없고, LoggerConfigModel.UseLog는 System이 기본값이면 false를 반환한다.
  • 운영 로그는 반대로 활성이다. UseOperationLog가 없으면 JobBase2와 AliveLogger는 기본적으로 운영 로그를 기록하는 쪽으로 동작한다.

가장 큰 차이: 프로젝트의 현재 App.config에는 RedisDistribute=false, MongoSaveData=false, UseOperationLog=false가 있지만, 업로드된 실행 config에는 이 키들이 없다. 따라서 실행 config 기준 분석 결과는 기존 보고서의 현재 프로젝트 config 기준과 다르다.

판단 설명 근거
객체 생성 MongoSaveData가 없어서 기본 true가 적용된다. 따라서 FIDType=PRICE에서도 MongoStockDataHandler 객체는 생성된다. SchedulerKiwoomReal.cs:164, KiwoomRedis.cs:80-85
저장 소비자 미시작 KiwoomRedis.Start()의 PRICE 경로에서 _mongoPriceHandler.StartAsync(...) 호출은 주석 처리되어 있다. 따라서 Mongo collection 생성/insert consumer는 시작되지 않는다. KiwoomRedis.cs:126-131, MongoStockDataHandler.cs:86-107

PG 전환 관점 이 실행 config의 KR/PRICE 경로만 보면 MongoDB는 실제 insert 경로가 막혀 있다. 하지만 객체 생성과 큐 적재는 살아 있으므로, PG 전환 전에 의도적으로 저장을 끈 것인지, 저장 재개를 예정한 코드인지 확인해야 한다. KiwoomRedis.cs:80-85, KiwoomRedis.cs:128-131

운영 로그 분석

  • UseOperationLog가 없으면 JobBase2._useOperationLog 기본값은 true다.
  • SchedulerKiwoomReal은 로드 성공 후 AliveLogger.Start()를 호출한다. AliveLogger도 UseOperationLog=false가 명시된 경우에만 중단한다.
  • 결론적으로 업로드 config 기준 운영 로그는 OperationTermHistoryBiz.USPOperationTermHistoryInsert를 통해 DB 기록을 시도한다. 이는 Kafka 로그와 별개다.

기존 프로젝트 App.config와의 차이

항목 프로젝트 App.config 업로드 실행 config 영향
RedisDistribute false 없음 → 기본 true 업로드 config 기준 Redis publish가 켜진다.
MongoSaveData false 없음 → 기본 true Mongo 핸들러가 생성된다. 단 실제 저장 consumer는 주석으로 미시작.
KiwoomStartDelay 중복 정의 있음 없음 → 기본 30 업로드 config에는 중복 키 오류가 없고, 시작 지연은 30분 기준이다.
UseOperationLog false 없음 → 기본 true 업로드 config 기준 운영 로그 DB 기록이 켜진다.
Redis endpoint Valkey dev 성격의 endpoint 기존 Redis/ElastiCache 성격의 endpoint Valkey 전환 분석은 업로드 config 기준 endpoint 교체 여부를 별도로 봐야 한다.
Kafka 관련 키 UseLog=false, LogCommunication=kafka 등 존재 없음 둘 다 런타임 Kafka 비활성으로 해석되지만, 이유가 다르다.

우선순위 리스크

순위 리스크 신뢰도 근거
1 업로드 config 기준 Redis publish가 실제로 켜진다. 높음 누락 키 기본값 true와 Redis handler 시작 코드가 직접 확인됨.
2 Mongo 저장은 의도와 달리 “켜진 것처럼 객체는 생성되지만 저장은 안 되는” 상태다. 높음 Mongo handler 생성 조건과 StartAsync 주석, AddPrice 호출이 함께 확인됨.
3 Mongo handler 큐가 drain되지 않아 메모리 증가 가능성이 있다. 중간-높음 BlockingCollection Add는 확인됨. 실제 이벤트량과 실행 시간에 따라 영향 크기는 달라진다.
4 Kafka는 런타임 미사용이지만 빌드/참조 의존성은 남아 있다. 높음 Core.Log 참조와 LoggerConfigModel/Logger의 비활성 조건이 확인됨.
5 운영 로그가 업로드 config 기준 활성화되어 DB 쓰기가 발생할 수 있다. 높음 UseOperationLog 누락 시 기본 true인 코드가 확인됨.

결론

업로드된 KiwoomReal 실행 config 기준으로 이 애플리케이션은 Kiwoom 실시간 PRICE 수집을 수행하고 Redis publish는 활성으로 동작한다. Kafka는 실행 경로에서 비활성으로 판단된다. MongoDB는 설정상 저장 의도가 켜진 것처럼 해석되지만, 실제 저장 consumer가 주석 처리되어 있어 저장은 되지 않고 큐만 쌓일 수 있는 구조가 가장 중요한 점이다.

따라서 Kafka 제거는 비교적 낮은 런타임 영향으로 보이나, Redis/Valkey 전환은 현재 실행 config에서 실제 사용 경로이므로 우선 검증 대상이다. MongoDB의 PG 전환은 먼저 현재 KR/PRICE에서 저장을 계속 막을 것인지, 아니면 PG 저장으로 재개할 것인지 결정해야 분석 범위가 확정된다.
