1. 레포 & 브랜치

레포스택브랜치 → 환경
java/Spring Boot (JDK, Gradle/Maven) + Dockerdevelop → dev, main → prod
frontend/Node.js + React (SSR)develop → dev, main → prod

bin/, checklists/, cs/, common-core/, summary/, summary2/, frontend2/, java2/ 는 분석/스크립트/레거시 자료. 현재 개발 대상 아님.

2. 배포 파이프라인 흐름

GitHub (java/, frontend/) │ │ push develop ─→ dev 빌드 │ push main ─→ prod 빌드 ▼ Jenkins (EC2 jenkins + agents) ─── Nexus(EC2) : 의존성 캐시 │ │ docker build ▼ ECR : 326539043145.dkr.ecr.ap-northeast-2.amazonaws.com/finup/<service> │ │ image tag = <env>-<git-shortsha>-<build-num> 예: dev-6d02661e-147 │ docker pull (대상 EC2가 Jenkins SSM 명령으로 pull & restart) ▼ EC2 (dev 서비스별 1대 / prod 서비스 묶음 HA) ─→ ALB ─→ 사용자
배포 규칙
  • prod 이미지는 대부분 IMMUTABLE — 같은 태그로 덮어쓸 수 없음. 롤백은 이전 태그로 재실행.
  • dev/prod 포트는 동일(Spring Boot 808x, React 300x). 환경 차이는 ALB 앞단에서 호스트로 구분.
  • Docker 이미지 크기: Spring Boot ≈ 270MB, React SSR ≈ 580MB.

3. 서비스 맵 (서비스 → 인프라)

각 서비스의 dev/prod 접속 URL, EC2, ECR, 포트를 한눈에 보는 지도. prod가 비어 있으면 아직 배포되지 않았거나 호스트 확인 필요.

6. 설정 / 환경변수 위치

현 시점 확인 필요
DB/캐시/외부 API 비밀은 AWS Secrets Manager / SSM Parameter Store 에 등록되어 있지 않음. 따라서 다음 중 하나로 추정:
  • Spring Boot: application-{env}.yml, 환경변수(도커 run 시 -e), 또는 Jenkins credentials 주입
  • React SSR: .env.{env}, 빌드 시 주입, 또는 PM2/pm2.config 환경변수
정확한 위치는 레포 java/, frontend/ 의 빌드/Docker 파일과 Jenkinsfile 확인 필요.

예외적으로 Secrets/SSM에 있는 것:

  • SSM finup-monitor-telegram-chat / finup-monitor-telegram-token — 운영 알림용
  • Secrets Manager: DataSync가 생성한 것만 있음 (앱 비밀 없음)

7.4 배포 (정상 경로)

  1. develop 또는 main 브랜치에 push
  2. Jenkins 해당 Job이 자동 트리거 (또는 수동 빌드)
  3. 빌드 성공 → ECR push → Jenkins SSM 명령으로 대상 EC2에서 pull & run
  4. ALB Target Health 가 healthy로 돌아오는지 확인

7.6 로컬 → dev 환경 디버깅

  • DB: VPN 접속 후 SQL Server 클라이언트로 maindb/predb 직접 접근
  • Redis/Valkey: VPN 접속 후 redis-cli -h <primary-endpoint> (포트 기본 6379)
  • API 목킹: dev ALB 호스트를 그대로 사용 가능 (`api-stock.internal.finup.co.kr`)
  • 로컬에서 dev Kafka 사용: MSK가 Private — 로컬 직접 연결 불가. SSM 포트포워딩 또는 점프 호스트 필요

8. 자주 묻는 질문

Q. 내 Spring Boot 서비스의 prod는 어디에 배포돼요?

A. 서비스 묶음에 따라 다릅니다. api-stock/finance/mentor/chat/admin 은 finup-api1/2 또는 finup-biz-api1/2 (biz 서비스인지 확인 필요), radar는 별도 finup-api-radar1/2, stockdata-core/realtime/admin 은 각각 전용 EC2(finup-stockdata-*1) — 3.4절 참조. 정확한 매핑은 Jenkins 배포 Job의 target host 리스트에서 확인 가능.

Q. 왜 prod ALB(alb-internal-api)는 API 호스트만 있어요? Front는?

A. alb-internal-api는 내부 API 라우팅 전용. Prod user-facing 트래픽(www.finup.co.kr, stock.finup.co.kr 등)은 현재 레거시 ELBMainExternal ALB 또는 CloudFront 를 경유하는 것으로 추정 — ELBMainExternal 리스너 Rule 상세 조사 필요.

Q. 포트 8083이 api-chat / api-mentor / pushapp 에 중복돼요?

A. 각각 다른 EC2에서 독립적으로 기동하므로 충돌 없음. ALB Target Group은 포트+인스턴스 매핑이라 같은 포트도 인스턴스별로 분리됩니다. 단, 한 EC2에서 여러 서비스를 같은 포트로 올리지는 못함.

Q. Valkey와 Redis 둘 다 있는데 뭘 써야 해요?

A. 신규 서비스는 Valkey (RG *-valkey-prod, *-valkey-dev). 레거시는 Redis RG (finup, stockdata, stockdatapubsub). 현재 stockdata 계열은 Valkey 전환 완료, finup Redis는 Valkey 대응본 미존재 — 전환 계획 확인 필요.

Q. Docker 이미지 빌드 실패 시 Nexus를 확인해야 하나요?

A. 예. Maven/Gradle 의존성은 Nexus(EC2) 미러에서 받습니다. Jenkins 빌드 에러가 could not resolve 또는 401 관련이면 Nexus (nexus.internal.finup.co.kr) 상태 확인.

Q. CloudWatch에 내 서비스 로그가 안 보여요.

A. 현재 애플리케이션 로그는 CloudWatch Logs로 전송되지 않습니다. SSM 접속 후 docker logs 로 확인. 중앙 집중이 필요하면 인프라 팀에 CloudWatch Agent/Fluent Bit 설정 요청.

Q. DB 비밀번호는 어디서 가져와요?

A. AWS Secrets Manager / SSM에 없습니다. Jenkins credentials 또는 팀 내 공유 설정 확인. 정확한 주입 경로는 Dockerfile / docker run 명령의 -e 또는 --env-file 인자를 따라가면 됩니다.
