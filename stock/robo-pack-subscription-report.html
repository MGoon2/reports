분석 범위: api-stock, front-stock 프로젝트

1. 문제 개요

문제: 신규 자동결제자 등록 시 카드 유효성 검증을 위한 100원 결제 로직이 누락되어 있음

영향: 카드 유효성 검증 없이 바로 정상 결제가 진행될 수 있어 결제 실패 리스크 증가

2. 기존 로직 설명 (레거시)

신규 자동결제자 등록 과정:

  • 카드의 문제가 없는지 확인하기 위해 100원의 결제 수행 (카드 유효성 검증)
  • 이 100원 결제가 성공적으로 이루어질 경우 정상 결제 진행
  • OrderAuto 테이블을 활용하여 자동결제 정보 관리

5. 100원 결제 로직 분석

5.1 레거시 코드 증거

파일: front-stock/src/app/payment/auto/card/issuebillkey/route.ts (라인 198)

// 레거시: ISBILL=N이면 AMOUNT 불포함 (주석처리됨)
// REQ_DATA.Add("AMOUNT", func.TEST_AMOUNT); // AUTH요청시 ISBILL=N인 경우 설정하지 않음.

분석:

  • 레거시 코드에서는 func.TEST_AMOUNT를 사용하여 100원 결제를 수행
  • 현재 구현에서는 이 부분이 주석 처리되어 있음
  • ISSUEBILLKEY 요청 시 AMOUNT 파라미터가 포함되지 않음

5.2 현재 구현 문제점

문제 1: 카드 유효성 검증 누락

  • 100원 결제로 카드 유효성을 검증하는 로직이 없음
  • 유효하지 않은 카드로 자동결제 등록이 가능할 수 있음
  • 이후 정기결제 배치에서 결제 실패가 발생할 수 있음

문제 2: ISBILL=N 설정의 부적절한 사용

  • AUTH 요청과 ISSUEBILLKEY 요청 모두 ISBILL=N으로 설정
  • ISBILL=N은 실제 결제를 하지 않는 인증만 수행한다는 의미
  • 카드 유효성 검증을 위한 소액 결제가 필요한 경우 ISBILL=Y로 설정해야 함

문제 3: OTBILL 호출 시점의 부적절성

  • 현재는 ISSUEBILLKEY 직후 바로 OTBILL 호출하여 전체 금액 결제
  • 100원 검증 결제 후 전체 금액 결제의 2단계 프로세스가 아님
  • 카드 유효성 검증 없이 바로 큰 금액 결제 시도

7. 정합성 문제점

7.1 데이터 정합성 이슈

  • OrderAuto 등록 시점: 카드 유효성 검증 전에 OrderAuto가 등록될 수 있음
  • 결제 상태 불일치: OrderAuto는 등록되었지만 실제 결제는 실패한 상태 발생 가능
  • 잔액 부족 처리: 카드 잔액 부족 시 OrderAuto 등록 후 결제 실패로 정리되지 않을 수 있음

7.2 트랜잭션 정합성 이슈

  • 외부 PG 호출: Danal OTBILL 호출은 DB 트랜잭션 외부에서 수행됨
  • 롤백 불가: OTBILL 성공 후 DB 실패 시 Danal CANCEL 호출로 보상하지만 완벽하지 않음
  • 중복 호출: PaymentBatchService에서 중복 호출 방지 로직이 있으나 완전하지 않을 수 있음

7.3 비즈니스 로직 정합성 이슈

  • 구독 시작 시점: 100원 검증 결제 성공 시점 vs 전체 금액 결제 성공 시점
  • 구독 기간: 구독 기간 시작 시점이 모호할 수 있음
  • 환불 처리: 100원 결제 실패 시 환불 로직이 필요할 수 있음

8. 문제 발생 시나리오

시나리오 1: 유효하지 않은 카드로 자동결제 등록

  1. 사용자가 만료된 카드로 자동결제 등록 시도
  2. 현재 로직: 100원 검증 없이 바로 전체 금액 결제 시도
  3. 결과: 결제 실패, OrderAuto는 등록된 상태로 남을 수 있음
  4. 영향: 사용자는 자동결제 등록 실패를 인지하지 못할 수 있음

시나리오 2: 잔액 부족 카드로 자동결제 등록

  1. 사용자가 잔액 부족 카드로 자동결제 등록 시도
  2. 현재 로직: 100원 검증 없이 바로 전체 금액 결제 시도
  3. 결과: 결제 실패, 사용자 경험 저하
  4. 영향: 100원으로 먼저 검증했다면 잔액 부족을 미리 알 수 있었음

시나리오 3: 정기결제 배치 실패

  1. 자동결제 등록 시 카드 유효성 검증 누락
  2. PaymentBatchScheduler가 매일 07:00에 정기결제 실행
  3. 유효하지 않은 카드로 인해 대량 결제 실패 발생
  4. 영향: CS 처리 부하 증가, 사용자 불만

public String procOtbill(AutoCardOtbillRequest req) {
    // 1. 자동결제 정보 조회 (BillKey 포함)
    AutoCardOrderAutoDetailResponse detail = getOrderAutoDetail(req.getUserIdx(), req.getOaIdx());
    
    // 2. Danal OTBILL 요청
    Map<String, String> otbillReq = new LinkedHashMap<>();
    otbillReq.put("AMOUNT", detail.getPrice());  // 전체 금액으로 바로 결제
    otbillReq.put("ISREBILL", "Y");
    otbillReq.put("BILLINFO", detail.getPayKey());
    
    Map<String, String> otbillRes = danalCreditClient.callCredit(otbillReq, danalCpid, danalKey, danalIv);
    
    if ("0000".equals(returnCode)) {
        // 3. 주문 등록
        orderInsertResult = procOrderInsert(insertReq);
        
        // 4. 성공/실패 이력 기록
        procOrderAutoSuccess(successReq);
    }
}

10. 개선 제안

10.1 100원 검증 결제 로직 추가

  1. ISSUEBILLKEY 요청 후 100원 결제 추가
  2. 100원 결제 성공 시에만 OrderAuto 등록
  3. 100원 결제 실패 시 사용자에게 명확한 에러 메시지 제공
  4. 100원 결제 성공 후 전체 금액 결제 진행

10.2 ISBILL 파라미터 조정

  1. AUTH 요청: ISBILL=N (유지 - 인증만 수행)
  2. ISSUEBILLKEY 요청: ISBILL=N (유지 - BillKey 발급만)
  3. 100원 검증 결제: ISBILL=Y, AMOUNT=100 (추가)
  4. 전체 금액 결제: ISBILL=Y, AMOUNT=전체금액 (유지)

10.3 트랜잭션 개선

  1. OrderAuto 등록을 100원 결제 성공 후로 이동
  2. 결제 실패 시 OrderAuto 롤백 로직 추가
  3. 중복 등록 방지 로직 강화

10.4 모니터링 강화

  1. 자동결제 등록 실패 로그 상세화
  2. 카드 유효성 검증 실패 모니터링
  3. 정기결제 배치 실패률 모니터링

11. 결론

핵심 문제

신규 자동결제자 등록 시 카드 유효성 검증을 위한 100원 결제 로직이 누락되어 있어, 유효하지 않은 카드로 자동결제가 등록될 수 있는 심각한 문제가 있습니다.

영향 범위

  • 사용자 경험 저하 (결제 실패)
  • CS 처리 부하 증가
  • 정기결제 배치 실패률 증가
  • 데이터 정합성 문제

조치 필요성

이 문제는 즉시 조치가 필요합니다. 100원 검증 결제 로직을 추가하여 카드 유효성을 사전에 확인함으로써 결제 실패를 방지하고 사용자 경험을 개선해야 합니다.
