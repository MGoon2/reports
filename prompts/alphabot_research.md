# Project Analysis Instructions

이 프로젝트는 C#/.NET 기반 윈도우앱이면서, 주기적으로 루프를 돌며 DB를 조회하고 반복 업무를 처리하는 레거시 프로젝트다.

Codex의 목표는 코드를 수정하는 것이 아니라, 이 프로젝트의 실행 구조, 반복 처리 흐름, DB 접근 방식, 운영 리스크, 리팩터링 후보를 분석하고 문서화하는 것이다.

---

## 1. 기본 원칙

- 사용자의 명시 요청 없이 소스코드를 수정하지 말 것.
- NuGet 패키지 업데이트, 프로젝트 파일 수정, 포맷팅 변경을 하지 말 것.
- 애플리케이션을 실행하지 말 것.
- 운영 DB, 운영 API, 외부 시스템에 접속하지 말 것.
- ConnectionString, 비밀번호, API Key, 토큰 값은 문서에 직접 기록하지 말 것.
- 민감 정보가 발견되면 값은 숨기고 파일 위치와 위험도만 기록할 것.
- 분석 결과는 가능한 한 파일명, 클래스명, 메서드명 단위로 근거를 남길 것.

---

## 2. 프로젝트 성격

이 프로젝트는 다음 성격을 동시에 가진다.

1. Windows Forms / WPF / 콘솔성 윈도우 실행 앱
2. 장시간 실행되는 운영 프로그램
3. 주기적으로 DB를 조회하는 반복 처리 프로그램
4. 특정 조건에 맞는 데이터를 찾아 업무를 자동 처리하는 배치성 프로그램
5. UI가 있더라도 실제 핵심은 백그라운드 루프와 DB 처리, 외부 API 호출 처리 등

따라서 UI 화면 구조보다 다음 항목을 더 중요하게 분석한다.

- 프로그램 시작 흐름
- 반복 루프 시작 위치
- DB 조회 주기
- 트랜잭션 범위
- 예외 처리 방식
- 중복 실행 방지 여부
- 로그 기록 방식
- 장시간 실행 시 안정성

---

## 3. 중점 분석 항목

### 3.1 프로그램 진입점

다음을 찾아 분석한다.

- `Program.cs`
- `Main()`
- `Application.Run(...)`
- 최초 실행 Form
- 초기화 메서드
- 설정 로딩 코드
- DB 연결 초기화 코드
- 타이머 또는 루프 시작 코드

분석해야 할 내용:

- 프로그램이 어떤 순서로 시작되는지
- 최초로 실행되는 Form 또는 Service가 무엇인지
- 시작 시 어떤 설정값을 읽는지
- 시작과 동시에 DB를 조회하는지
- 시작 실패 시 어떻게 처리되는지

### 3.2 UI와 백그라운드 처리 관계

윈도우앱이므로 UI와 업무 처리 로직의 관계를 분석한다.

찾아야 할 항목:

- 버튼 클릭 이벤트에서 배치 작업을 시작하는지
- Form Load 이벤트에서 자동으로 루프를 시작하는지
- UI Thread에서 DB 작업을 직접 수행하는지
- `BackgroundWorker`, `Task`, `Thread`, `Timer`를 사용하는지
- 장시간 작업으로 UI가 멈출 가능성이 있는지
- 사용자가 중지 버튼을 누를 수 있는 구조인지
- 종료 시 루프가 안전하게 중단되는지

주의해서 봐야 할 위험:

- `Form_Load`에서 바로 무한 루프 실행
- UI Thread에서 `Thread.Sleep()` 사용
- 버튼 중복 클릭으로 작업 중복 실행
- 종료 버튼을 눌러도 백그라운드 작업이 계속 도는 구조
- 예외 발생 시 UI에는 정상처럼 보이지만 내부 루프가 죽는 구조

---

### 3.3 반복 루프 / 배치 처리 구조

이 프로젝트의 핵심 분석 대상이다.

다음을 반드시 찾는다.

- `while`
- `for`
- `foreach`
- `Thread.Sleep`
- `Task.Delay`
- `System.Threading.Timer`
- `System.Windows.Forms.Timer`
- `System.Timers.Timer`
- `BackgroundWorker`
- `Task.Run`
- `new Thread`
- 재귀적으로 반복 호출되는 메서드
- 일정 시간마다 DB를 조회하는 메서드

분석해야 할 내용:

- 루프 시작 위치
- 루프 종료 조건
- 루프 실행 간격
- 루프 1회당 수행 작업
- 루프 내부 DB 조회 횟수
- 루프 내부 DB 저장 횟수
- 루프 내부 외부 API 호출 여부
- 루프 중 예외 발생 시 계속 실행되는지
- 루프가 중복으로 실행될 수 있는지
- 루프 상태를 확인할 수 있는 로그가 있는지

특히 다음 위험을 확인한다.

- 종료 조건 없는 무한 루프
- 너무 짧은 sleep 간격
- DB 조회 실패 시 즉시 재시도하는 구조
- 예외 발생 후 조용히 루프가 종료되는 구조
- 여러 루프가 같은 데이터를 동시에 처리하는 구조
- 처리 완료 표시 전에 장애가 나면 중복 처리되는 구조
- 처리 완료 표시 후 장애가 나면 데이터 유실되는 구조

### 3.4 DB 접근 분석

다음을 찾아 분석한다.

- `SqlConnection`
- `SqlCommand`
- `SqlDataReader`
- `SqlDataAdapter`
- `DataSet`
- `DataTable`
- `ExecuteReader`
- `ExecuteNonQuery`
- `ExecuteScalar`
- Stored Procedure 호출
- 직접 SQL 문자열 조립
- 트랜잭션 사용
- ConnectionString 사용
- 공통 DB Helper 클래스

분석해야 할 내용:

- DB 접속이 어디에서 발생하는지
- DB Helper가 있는지
- 커넥션이 매번 새로 열리는지
- 커넥션이 재사용되는지
- `using`으로 안전하게 닫히는지
- 트랜잭션 범위가 너무 넓지 않은지
- 반복문 안에서 DB를 과도하게 호출하는지
- SQL 문자열을 직접 이어붙이는지
- Stored Procedure 호출 구조가 일관적인지
- CommandTimeout 설정이 있는지
- 장애 발생 시 롤백되는지

특히 다음 위험을 확인한다.

- Connection Dispose 누락
- Reader Close 누락
- Transaction Commit/Rollback 누락
- 반복문 내부 개별 쿼리 과다 호출
- 장시간 열린 Transaction
- `catch`에서 예외를 무시하고 계속 진행
- SQL Injection 가능성
- 운영 ConnectionString 하드코딩
- DB 장애 시 무한 재시도

### 3.5 데이터 처리 흐름

반복 루프에서 처리하는 업무 흐름을 추적한다.

분석해야 할 내용:

- 어떤 조건의 데이터를 조회하는지
- 조회한 데이터를 어떤 기준으로 처리하는지
- 처리 전/후 상태값이 어떻게 바뀌는지
- 실패한 데이터는 어떻게 표시하는지
- 재처리 가능한 구조인지
- 중복 처리를 막는 컬럼이나 상태값이 있는지
- 처리 순서가 중요한지
- 여러 건 처리 중 일부 실패 시 어떻게 되는지

가능하면 다음 형식으로 정리한다.

```text
조회 대상 데이터
→ 처리 조건
→ 처리 메서드
→ 성공 시 상태 변경
→ 실패 시 상태 변경
→ 로그 기록
→ 다음 루프 대기
